# 뽀모올리 — PRD

> 앱 이름 확정: **뽀모올리** (캐릭터: 뽀모=토마토, 올리=올리브, 2026-07-16)

- **작성일**: 2026-07-16 (사용자 승인: 2026-07-16)
- **한 줄 정의**: 파스타 종류를 고르면 딱 맞는 삶기 타이머가 돌아가고, 다 삶으면 그 파스타로 만들 맛있는 레시피가 나오는 앱

## 문제와 타겟
파스타마다 알맞은 삶는 시간이 달라 매번 봉지 뒷면을 찾아보거나 감으로 삶는다.
타겟: 집에서 파스타를 해 먹는 요리 초보~중수.

## 핵심 사용자 시나리오
1. 파스타 선택 (대표 면 10종으로 시작: 스파게티, 링귀네, 페투치네, 펜네, 푸실리, 파르팔레, 리가토니, 카펠리니, 오레키에테, 마카로니)
2. 파스타 상세 화면 하나에서 전부 해결 (2026-07-16 사용자 확정):
   - **상단: 타이머 고정(sticky)** — 알덴테/기본 토글, "봉지 시간 직접 입력", 시작 버튼.
     스크롤해도 타이머는 계속 보인다.
   - **하단: 그 파스타의 레시피 1~2개** — 면 삶는 동안 스크롤하며 소스를 준비하는 흐름.
3. 물 끓고 면 넣는 순간 시작 → 완료 시 소리+진동+알림 (레시피 보던 중에도 즉시 인지)

## 기능 목록
### MVP (이것만 되면 런칭)
- [ ] 파스타 10종 데이터 (종류별 표준 시간 + 알덴테 시간)
- [ ] 타이머: 시작/일시정지/리셋, 화면 꺼짐 방지(Wake Lock), 백그라운드 시간 보정, 완료 시 소리+진동+알림
- [ ] 커스텀 시간 입력 (봉지 표기 시간)
- [ ] 파스타별 레시피 1~2개 (Claude가 표준 조리법 기반, 한국 가정 재료 기준으로 오리지널 작성)
- [ ] 상세 화면 레이아웃: 상단 고정 타이머 + 하단 레시피 스크롤 (한 화면)
- [ ] PWA (홈 화면 추가, 오프라인 동작)
- [ ] 주방 UX: 큰 버튼, 큰 글씨 (손에 물 묻은 상태 가정)

### 이후 (런칭 뒤에)

- [ ] 즐겨찾기·최근 사용
- [ ] 포트폴리오 사이트(`~/project/portfolio-site`)에 사례 등록

### 하지 않기로 한 것
- **수익화 전면 제외** (2026-07-17 사용자 결정) — 광고도, 쿠팡 파트너스도 안 붙인다.
  이 앱의 목적은 돈이 아니라 **포트폴리오**다. 수익을 안 내면 앱인토스 정산의
  사업자등록 요건도 무관해진다.
- 재료 데이터의 구조화(이름/양 분리)는 그대로 유지 — 파트너스 대비가 아니라
  데이터 위생 차원에서 옳고, 나중에 마음이 바뀌어도 열려 있다.

## 아키텍처 결정
- **Next.js (App Router) + Vercel, PWA** — 워크스페이스 표준 스택. 무료 vercel.app으로 런칭.
- **서버·DB 없음** — 파스타 시간표·레시피는 정적 데이터 파일. 유지비 0원.
- **레시피 재료는 구조화 데이터로 저장** (이름/양/단위 분리) — 나중에 쿠팡 파트너스
  링크를 재료 단위로 끼우는 작업이 데이터 파일 수정만으로 되게.

## 목적과 성공 기준

**목적: 포트폴리오** (2026-07-17 확정). 수익이 아니라 "Product PM이 기획부터 디자인·개발·
출시까지 혼자 끝냈다"는 증거물. 사용자의 기존 포트폴리오 사이트 포지셔닝
("직접 빌드하는 Product PM")과 정확히 맞물린다.

- **1차**: 사용자 본인이 파스타 삶을 때 실제로 켠다 (안 쓰는 앱은 포트폴리오로도 약하다)
- **2차**: 지인 공유 후 재사용이 관찰된다
- **3차**: 앱인토스 출시 — "토스 앱에 출시된 미니앱"이라는 이력 자체가 자산
- **부가 자산**: 결과물뿐 아니라 **과정**이 포트폴리오다 — 디자인 시스템 수립,
  3모델 교차검증 네이밍, 한국어 문장 검수 파이프라인, 3방향 자동 검수.
  PM 포지션에는 앱 자체보다 이 의사결정 기록이 더 강력할 수 있다.
