# 파스타 타이머 (뽀모올리)

**면 종류를 고르면 딱 맞는 타이머가 돌아가고, 삶는 동안 그 파스타의 레시피를 볼 수 있는 앱.**

기획·디자인·개발·출시를 1인이 수행한 사이드 프로젝트입니다.

## 왜 만들었나

파스타는 면 종류마다 삶는 시간이 다릅니다. 스파게티 8분, 카펠리니 2분, 리가토니 11분.
그런데 실제 상황에서는 이렇게 됩니다.

1. 봉지 뒷면의 작은 글씨를 찾아 읽는다 (손에 물이 묻은 채로)
2. 폰 기본 타이머를 켜서 숫자를 직접 입력한다
3. 면이 삶아지는 8분 동안 뭘 해야 할지 몰라 그냥 기다린다

세 번째가 핵심입니다. **면이 삶아지는 시간은 소스를 만드는 시간**인데, 대부분의 타이머 앱은
숫자만 세고 끝납니다. 이 앱은 그 8분을 채웁니다.

## 핵심 설계 결정

| 결정 | 이유 |
|---|---|
| **타이머와 레시피를 한 화면에** | 완료 후 레시피를 제안하면 이미 늦습니다. 상단에 타이머를 고정하고 아래로 레시피를 두면, 면 삶는 동안 스크롤하며 소스를 준비하는 실제 요리 순서와 일치합니다. |
| **네이티브 앱 대신 PWA** | 심사·연회비 없이 즉시 배포하고, 홈 화면에 추가하면 앱처럼 동작합니다. 검증 후 앱스토어로 확장 가능. |
| **서버·DB 없음** | 파스타 시간표와 레시피는 정적 데이터. 운영비 0원, 100% 정적 빌드. |
| **수익화 전면 제외** | 광고도 제휴 링크도 붙이지 않았습니다. 목적이 수익이 아니라 포트폴리오이고, 주방에서 쓰는 도구에 광고가 끼면 도구로서 나빠집니다. |
| **타이머 숫자를 세리프체로** | 타이머 앱은 관행적으로 기계적인 모노스페이스를 씁니다. 의도적으로 반대로 가서, 스크린샷만 봐도 이 앱임을 알 수 있게 했습니다. |

## 사용 맥락이 만든 제약

타겟은 **주방에 서 있는 사람**입니다. 여기서 세 가지 규칙이 도출됐습니다.

- **터치 타깃 최소 48px** — 손에 물이 묻어 있고 조준이 정확하지 않습니다
- **본문 15px 하한** — 조리대에서 팔 길이만큼 떨어져 봅니다
- **완료 1분 전 예비 신호** — 레시피 대부분이 면수를 쓰는데, 알람이 울린 뒤엔 이미 손이 바쁩니다

초보가 파스타를 망치는 1순위는 시간이 아니라 **물이 적어 면이 붙는 것과 소금 누락**입니다.
그래서 타이머 시작 전에 물·소금 기준을 먼저 보여줍니다.

## 만들면서 막혔던 것

- **8GB RAM 맥에서 iOS 시뮬레이터가 죽음** — 부팅 직후 CoreSimulator가 종료됐습니다.
  시뮬레이터 검증을 포기하고 실기기 배포 검증으로 전환했습니다. 어차피 진동은 시뮬레이터가
  재현하지 못하므로 실기기가 더 정확한 검증입니다.
- **앱인토스는 SSR 금지** — 토스 미니앱 출시 요건을 확인하니 서버 렌더링이 막혀 있었습니다.
  다행히 초기에 "서버·DB 없음"을 택한 덕에 `output: "export"` 전환만으로 통과했습니다.
- **접근성 대비 미달** — 주 액션 버튼이 WCAG AA 기준(4.5:1)에 못 미치는 3.55:1이었습니다.
  색 인상을 유지한 채 팔레트 안에서 4.86:1로 조정했습니다.

## 기술

- **Next.js 15** (App Router) · TypeScript · 정적 export
- **PWA** — 홈 화면 추가, 서비스워커 오프라인 캐시
- 타이머는 `setInterval` 누적이 아닌 **timestamp 기반** — 백그라운드에 다녀와도 시간이 어긋나지 않습니다
- Wake Lock(화면 꺼짐 방지) · Web Audio(완료음) · Vibration · Notification
- 의존성 최소 — 상태 관리 라이브러리, UI 프레임워크 없음

```bash
bun install
bun run dev          # 개발 서버
bun test             # 타이머 계산 로직 테스트
bun run build        # 정적 빌드 → out/
NEXT_PUBLIC_TARGET=toss bun run build   # 앱인토스 미니앱 빌드
```
